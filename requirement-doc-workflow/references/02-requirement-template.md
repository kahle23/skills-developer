# 02 — 需求文档模板

> 面向受众：需求方（业务/产品）。语言要**说人话**，技术概念要翻译成业务语言。

文档必须包含以下七个章节。每章末尾有 **mini-gate**（章节级检查点），写完一章过一遍。

---

## 文档头信息

```
需求名称：{XXX 功能}
需求提出人：{姓名/角色}
提出日期：{YYYY-MM-DD}
需求编号：{如有，如 PRD-2026-001}
需求优先级：{高/中/低}
当前状态：{草案 / 已确认 / 已进入设计}
```

---

## 一、需求背景与目标

### 1.1 背景

为什么要做这个需求？回答"现在遇到了什么问题"或"什么变化驱动了这个需求"。

- 当前痛点（如有）：现在流程哪里不顺、哪里靠人工、哪里容易出错
- 触发事件（如有）：业务变化、合规要求、用户反馈

### 1.2 目标

做完之后想达成什么效果？回答"做成之后会变怎样"。

- 核心目标（必写）：一句话说清最重要的业务目标
- 次要目标（如有）：附带想实现的

### 1.3 名词定义

本需求涉及的专有名词（业务术语），用需求方能懂的语言解释。

| 术语 | 含义 |
|------|------|
|      |      |

**mini-gate**：

- [ ] 背景说清了"为什么做"，不是功能名的复述
- [ ] 目标可衡量或可感知（"减少人工核对工作"而非"提升体验"）
- [ ] 有专有术语的，都已定义

---

## 二、范围边界

### 2.1 本次包含

列出本次需求**要做的**功能点清单（一句话级，先列要点，第四章展开细节）：

1. {功能点 1}
2. {功能点 2}
3. ...

### 2.2 本次不包含

明确列出**不做的**，至少 2 项：

1. {排除项 1}——原因：{为什么不这次做}
2. {排除项 2}——原因：{原因}

> ⚠️ "不做什么"和"做什么"一样重要。显式排除能防止后续范围蔓延。

### 2.3 涉及的用户角色

| 角色 | 说明 | 本需求中的主要操作 |
|------|------|------------------|
|      |      |                  |

**mini-gate**：

- [ ] 包含项和排除项都列了
- [ ] 排除项至少 2 项且有原因
- [ ] 用户角色已列出

---

## 三、业务流程

用需求方能看懂的方式描述核心业务流程。

### 3.1 流程概述

用文字或简单流程图说明"从头到尾怎么走"：

```
{角色A} 做 {动作1} → 系统 {处理} → {角色B} 做 {动作2} → 结果
```

> 如果流程复杂，画简单的流程图（方框+箭头），不要画技术架构图。

### 3.2 分支说明

流程中的关键分支和判断点：

| 判断点 | 条件 | 走向 |
|--------|------|------|
|        |      |      |

**mini-gate**：

- [ ] 核心流程能从头走到尾，无断点
- [ ] 关键分支已列出
- [ ] 流程图/文字描述需求方能看懂（无技术术语）

---

## 四、功能清单

这是文档的核心。逐个功能点详细说明，**每个功能点必须带验收标准**。

### 4.1 功能概览

| 序号 | 功能点 | 优先级 | 涉及角色 |
|------|--------|--------|---------|
| 1    |        | 高/中/低 |        |

### 4.2 逐功能点说明

#### 功能点 1：{功能名称}

**触发条件**：用户在什么情况下用这个功能。

**操作步骤**：

1. 用户 {操作}
2. 系统 {响应}
3. ...

**验收标准**（至少 2 条，必须可验证）：

- ✅ {操作后}，能看到 {具体结果}——验证方法：{怎么看}
- ✅ {边界场景}时，系统 {预期行为}——验证方法：{怎么看}

**异常处理**：

- {异常场景}时，系统 {预期行为}（如提示什么、是否回滚）

> 验收标准写法参考：用"用户做了 X，系统应该 Y"的句式。避免"功能正常""体验良好"这种不可验证的描述。

#### 功能点 2：{功能名称}

（同上结构）

**mini-gate**：

- [ ] 每个功能点都有触发条件
- [ ] 每个功能点至少 2 条可验证的验收标准
- [ ] 主要异常场景已列出
- [ ] 优先级已标注

---

## 五、数据预期

> 这一章不写表结构（那是设计文档的事），只写需求方关心的数据层面的事。

### 5.1 需要记录什么

这个功能运行过程中，需要记住哪些信息（从业务角度）：

- {数据项 1}：为什么需要
- {数据项 2}：为什么需要

### 5.2 数据来源

| 数据 | 从哪来 | 备注 |
|------|--------|------|
|      | 用户填写 / 系统生成 / 从XX模块获取 |      |

### 5.3 数据去向

这些数据做完之后给谁看、用在哪：

- {数据} → {用途}

**mini-gate**：

- [ ] 核心数据项已列出（业务视角，不是字段名）
- [ ] 数据来源已标注（手填/自动/复用）
- [ ] 不涉及敏感数据流转（如涉及，标注权限要求）

---

## 六、非功能需求

> 不是每个需求都有，有就写，没有可以写"无特殊要求"。

- **性能**：{如：列表要在 2 秒内加载、支持 100 人同时操作}
- **兼容性**：{如：要支持手机端、要支持 IE11}
- **安全/权限**：{如：只有管理员能看、涉及个人隐私数据需脱敏}
- **可用性**：{如：7x24 可用、允许每周三维护窗口}
- **合规**：{如：需满足等保要求、数据保留 3 年}

**mini-gate**：

- [ ] 有特殊非功能要求的，已列出
- [ ] 没有特殊要求的，已标注"无特殊要求"

- [ ] 所有 `[待确认]` 项要么已解决，要么已转为开放问题
