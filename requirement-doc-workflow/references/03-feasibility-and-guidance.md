# 03 — 可行性校验与引导话术

> ⭐ 这是本技能的核心价值所在。普通文档生成只管"把需求写下来"，本技能要**判断需求能不能做、引导需求方往更优方向走**。

## 核心原则回顾

> **引导，不是驳回。** 当需求方提出的需求难以实现时，不要说"做不到"。理解目标 → 摸清能力 → 提出替代 → 讲清权衡。

---

## 一、可行性三级判定

对功能清单中的每个功能点，结合阶段 2 摸到的项目现状，判定属于哪一级：

### ✅ 直接实现

**判定标准**：

- 现有模块能力可直接支撑
- 或现有模式有先例可循（如项目已有类似的增删改查）
- 实现成本低，不需要新建基础设施

**处理**：直接纳入需求文档，无需特别引导。

**示例**：

> 需求："盘点结果要能导出 Excel"
> 项目现状：已有资产列表导出 Excel 的功能
> 判定：✅ 直接实现（复用现有导出能力）

### 🔶 引导调整

**判定标准**：

- 现有能力部分支撑，但需求方的设想方案不是最优
- 有业务等效但成本更低的替代方案
- 需求方的预期效果和现有能力有偏差

**处理**：执行引导话术四步法，和需求方协商调整。

**示例**：

> 需求："盘点时要实时扫码，扫一个自动入库"
> 项目现状：系统是 Web 端，无扫码组件，但有批量导入 Excel 功能
> 判定：🔶 引导调整
> 引导：理解目标是"快速录入盘点结果" → 说明现有能力是 Web 端批量导入 → 提出替代"导出现有清单→线下盘点→批量导入" → 讲清权衡（实时扫码需开发移动端，工作量大 3 倍；批量导入现在就能用）

### 🔴 需协商

**判定标准**：

- 现有能力完全不支持，需新建基础设施（如项目是 Web 端，需求要原生 App 功能）
- 涉及底层架构改动
- 需求方可能低估了成本

**处理**：明确告知成本和风险，给出信息，**让需求方决定是否保留**。不要替需求方拍板删需求。

**示例**：

> 需求："盘点要能像微信一样实时多人协同编辑"
> 项目现状：传统 CRUD 系统，无协同编辑能力
> 判定：🔴 需协商
> 处理：说明"实时协同"需要 WebSocket+冲突解决机制，属于基础设施级改动 → 提供降级方案（加锁编辑+刷新提示）→ 由需求方决定

## 二、引导话术四步法

对 🔶 和 🔴 的功能点，按以下四步和需求方沟通：

### 第 1 步：复述业务目标

先确认理解对了需求方想解决什么问题。**这一步防止你提出的替代方案跑偏。**

话术模板：

> "我先确认一下你想达成的效果：你的目标是 {用业务语言复述}，对吗？"

> ⚠️ 复述业务目标，不是复述技术方案。需求方说"我要扫码"，业务目标是"快速录入"，不要复述成"你要扫码功能"。

### 第 2 步：说明现有能力

客观说明项目现在能做到什么程度，不夸大不贬低。

话术模板：

> "目前项目在这一块的能力是：{列举现有能力}。能做到 {X}，但暂时没有 {Y}。"

> 用业务语言，不堆技术术语。不要说"我们没有 WebSocket 长连接支持"，要说"目前系统不支持多人同时编辑同一份数据"。

### 第 3 步：提出替代方案

给出业务等效但更易实现的方案。**至少给一个，最好给两个让需求方选。**

话术模板：

> "有一个效果差不多但更轻量的做法：{方案 A}。另外也可以考虑 {方案 B}。"

替代方案思路参考下一节的「替代方案速查表」。

### 第 4 步：讲清权衡

把不同方案的代价和效果用需求方能理解的方式说清楚。

话术模板：

> "两个方案的区别在于：方案 A {优点}，但 {缺点}；方案 B {优点}，但 {缺点}。如果用你最初设想的方案，需要 {大致工作量/周期}，而且 {风险}。你看怎么选？"

> 讲成本用相对量（"大概 3 倍的工作量""可能要拖两周"），不用绝对数字（除非你真的能估准）。

## 三、替代方案速查表

常见的"需求设想 → 更优替代"对照：

| 需求方的设想 | 现有能力常见情况 | 引导的替代方向 |
|-------------|----------------|--------------|
| 实时扫码录入 | Web 端无扫码 | 导出清单→线下盘点→批量导入；或后续做小程序扫码 |
| 多人实时协同编辑 | 传统 CRUD 无协同 | 加锁编辑+其他人刷新提示；或分配不同子任务各自编辑 |
| 智能推荐/自动判断 | 无 AI 能力 | 规则配置（条件→动作）；或人工标记后系统筛选 |
| 实时消息推送 | 无推送通道 | 站内消息+定期邮件汇总；或对接已有企业微信/钉钉 |
| 复杂自定义报表 | 仅有固定报表 | 预置几种常用报表模板+筛选条件；复杂分析导出原始数据 |
| 手写签名 | Web 端无签名组件 | 上传签名图片；或打印签字后拍照上传 |
| 地图实时定位 | 无地图能力 | 手动填写位置；或定时上报经纬度 |
| 复杂审批流引擎 | 仅有固定审批 | 预置几种审批模板；或支持增删审批节点 |
| 大数据量实时统计 | 无大数据能力 | T+1 离线统计；或定时任务预计算 |
| 富文本/复杂排版 | 仅有纯文本 | Markdown 输入；或对接已有文档编辑器 |

> 这些只是常见模式。实际判定要结合项目摸底结果，不能生搬。

## 四、成本提示语

引导时如何体面地让需求方意识到成本：

### 不要说的 ❌

- "这个做不了"
- "技术实现不了"
- "这个太复杂了"
- "要花很长时间"

这些话会让需求方觉得你在推活。

### 可以说的 ✅

- "这个可以做，但需要先搭一些基础能力，周期会拉长。有个更快的做法是……"
- "现有系统这块是 {现状}，要做到你说的效果，改动范围比较大。我们可以先用 {替代方案} 满足核心需求，后续再迭代。"
- "这个效果很理想，不过当前优先把 {核心目标} 做出来更重要。这个可以放到二期。"
- "我了解下来，要完整实现需要涉及 {大致范围} 的改动。你看看是这次一起做，还是分阶段？"

**核心思路**：永远先给"能做"的态度，再用"成本/周期/范围"引导需求方权衡，而不是直接拒绝。
