# 01 — 工作流

本工作流分 **6 个阶段、5 道门禁（Gate）**。每个阶段的 Gate 检查通过后才能进入下一阶段。

如果 Gate 没过，停在这一步补齐，不要往后跑。

---

## 阶段 1：输入收集

**目标**：搞清楚需求方到底想要什么，以及手头有哪些信息可以用。

**动作**：

### 1a. 收集需求方的原始输入

从以下渠道获取需求方的原始表达：

- 用户直接在对话中描述的需求
- 用户提供的截图 / 原型图（分析图片内容）
- 用户提供的文档片段（需求草稿、会议纪要等）

> 即使只有一句话（如"做个资产盘点"），也先记录下来，进入阶段 2 摸底后再回头澄清。

### 1b. 确认项目背景来源

确认背景知识从哪来（参考 SKILL.md 的「项目背景知识来源」）：

- 用户是否启用了项目知识库技能？→ 读取项目背景
- 用户指定的工作目录里有什么？→ 扫描代码和文档
- 都没有？→ 标记"缺少项目背景"，Gate2 时告知用户

### Gate 1：输入齐全确认

**过门检查清单**：

- [ ] 已记录需求方的原始表达（哪怕是一句话）
- [ ] 已确认是否有截图/原型/文档附件
- [ ] 已确认项目背景来源（知识库 / 工作目录 / 仅口头）
- [ ] 已识别需求中提到的核心业务对象（如"资产""盘点""审批"）

**没过 Gate 1 的典型情况**：

- 需求方说得很模糊（"优化一下系统"）→ 追问："优化哪一块？遇到了什么问题？"
- 完全没有项目背景 → 问："这个需求是针对哪个系统/模块的？有项目文档或代码可以参考吗？"

---

## 阶段 2：现状摸底

**目标**：搞清楚项目里已有什么能力、什么模块、什么约定，为后续可行性判断打基础。

**动作**：

### 2a. 摸项目能力（有什么）

根据阶段 1 确认的背景来源：

| 来源 | 怎么摸 |
|------|--------|
| 知识库技能 | 读取项目概况、模块清单、业务术语表、已有功能说明 |
| 工作目录代码 | 扫目录结构找模块划分；读 README、docs/ 下的文档；看路由/Controller 找已有接口 |
| 工作目录文档 | 读取 docs/、需求文档目录下的历史文档，了解文档风格和已有需求 |

重点摸清：

- **已有模块清单**：项目现在有哪些功能模块？（判断新需求和现有模块的关系）
- **业务术语**：项目里有哪些专有名词？（如"资产编码""盘点单""审批流"）
- **技术栈线索**：前端什么框架、后端什么框架？（影响可行性判断）

### 2b. 摸相关模块（新需求靠近谁）

新需求大概率不是凭空出现的，找到它"挂靠"的现有模块：

- 需求涉及"资产"→ 找项目里已有的资产相关模块
- 需求涉及"审批"→ 找项目里已有的审批流实现
- 需求涉及"报表"→ 找项目里已有的报表/导出功能

> 目的：判断新需求能在多大程度上复用现有能力，以及会碰到什么约束。

### Gate 2：现状已摸清

**过门检查清单**：

- [ ] 已了解项目的模块清单（或确认无法获取，已告知用户）
- [ ] 已识别新需求挂靠的现有模块（或确认是全新领域）
- [ ] 已了解相关模块的现有能力边界
- [ ] 如果缺少项目背景，已向用户说明"可行性判断可能不准"

**没过 Gate 2 的典型情况**：

- 知识库/代码读不到有效信息 → 问："能指一下相关代码或文档在哪个目录吗？"
- 找到多个可能相关的模块，不确定哪个 → 问："这个需求和 XX 模块还是 YY 模块关系更大？"

---

## 阶段 3：需求澄清

**目标**：通过结构化提问，把模糊需求逼问到清晰。**这是本技能最耗精力的阶段。**

**动作**：

### 3a. 对照问题库提问

读取 `04-clarification-questions.md`，按以下策略提问：

1. **先过通用七维度**：范围 / 用户 / 流程 / 数据 / 权限 / 异常 / 优先级
2. **再过功能类型专项问题**：根据需求类型（增删改查/审批流/报表/导入导出/消息通知）选对应问题

> ⚠️ **提问节奏控制**：不要一次抛出 20 个问题。每轮问 3-5 个最关键的，拿到答案再问下一批。详见 04 的「提问节奏」小节。

### 3b. 边问边记录

每得到一个答案，立即记录到需求草稿中：

- 明确的需求 → 记入"功能清单"
- 明确的排除 → 记入"不做什么"
- 暴露的约束 → 记入"约束条件"
- 需求方也不确定的 → 标记 `[待确认]`，继续往下问

### Gate 3：需求已澄清

**过门检查清单**：

- [ ] 能用一句话说清这个需求要实现什么业务目标
- [ ] 能列出"做什么"（功能点）和"不做什么"（至少 2 项排除）
- [ ] 每个功能点有明确的触发条件和预期结果
- [ ] 至少识别出 1 个异常场景或边界条件
- [ ] 核心用户角色已明确
- [ ] 优先级已初步排序（高/中/低）

**没过 Gate 3 的典型情况**：

- 功能点还是太泛（"管理资产"）→ 追问到具体操作（"录入、查询、修改、删除，哪些需要？"）
- 说不清验收标准 → 问："做完之后，你怎么验收？看到什么算成功？"
- 角色不清 → 问："谁用这个功能？只有管理员还是普通员工也能用？"

---

## 阶段 4：可行性校验

**目标**：结合阶段 2 摸到的项目现状，判断需求中的每个功能点是否可实现、实现成本如何。**不合理的，引导调整。**

> ⭐ 这是本技能的核心价值环节。详见 `03-feasibility-and-guidance.md`。

**动作**：

### 4a. 逐功能点做可行性判定

对功能清单中的每个功能点，判定属于哪一级：

| 判定 | 含义 | 处理 |
|------|------|------|
| ✅ 直接实现 | 现有能力可支撑，成本低 | 直接纳入需求文档 |
| 🔶 引导调整 | 现有能力部分支撑，需调整方案 | 用引导话术和需求方协商替代方案 |
| 🔴 需协商 | 现有能力不支持，需新建基础设施或大改 | 明确告知成本，由需求方决定是否保留 |

### 4b. 对 🔶 和 🔴 的功能点执行引导

执行引导话术四步法（详见 03）：

1. 复述需求方想达成的业务目标
2. 说明现有能力到哪一步
3. 提出业务等效的替代方案
4. 用业务语言讲清权衡（工作量/效果/风险）

> 不要替需求方做决定。给出信息和建议，让需求方拍板。

### Gate 4：可行性已校验

**过门检查清单**：

- [ ] 每个功能点都做了可行性判定（✅/🔶/🔴）
- [ ] 所有 🔶 和 🔴 的功能点都已与需求方沟通
- [ ] 需求方对调整方案有明确反馈（接受/拒绝/待定）
- [ ] 没有遗留的"不知道能不能做"的功能点

**没过 Gate 4 的典型情况**：

- 有功能点没判定 → 回去补判定
- 需求方对替代方案没表态 → 问："这个调整你接受吗？还是坚持原方案？"

---

## 阶段 5：文档编写

**目标**：按模板输出完整的需求文档。

**动作**：

按 `02-requirement-template.md` 的七章节结构输出文档。模板每章末尾有 mini-gate。

### Gate 5：文档完整性检查

**过门检查清单**：

- [ ] 七个章节全部输出
- [ ] 每章的 mini-gate 检查通过
- [ ] 所有 Gate 3 中标记的 `[待确认]` 项已解决或明确标注为开放问题
- [ ] 每个功能点都有验收标准（可验证，不是"完成该功能"）
- [ ] 可行性校验中的 🔶/🔴 项在文档中已有结论说明

**没过 Gate 5 的典型情况**：

- 验收标准太空（"功能正常"）→ 改成具体的（"提交后列表刷新，状态变为'已提交'"）
- `[待确认]` 还没清 → 列出来让需求方逐个确认

---

## 阶段 6：交付确认

**目标**：文档定稿，需求方确认可以进入下一环节（设计/开发）。

**动作**：

1. 向需求方展示文档摘要：功能点数量、验收标准数量、开放问题数量
2. 逐项确认 Gate 5 的检查结果
3. 根据反馈迭代调整

### Gate 6：交付确认

**过门检查清单**：

- [ ] 需求方确认文档准确反映了需求
- [ ] 开放问题清单已列出（即使没全部解决）
- [ ] 文档已保存到正确位置
- [ ] 建议下一步：交接给 `feature-design-workflow` 进入技术设计

---

## 草台班子速查：只有一句话需求时怎么办

团队实际场景：需求方丢来一句话（"做个盘点功能"）或一张截图，没有正式需求说明。

**应对策略**：

1. **先摸底再追问**：不要立刻追问细节。先进入阶段 2 摸清项目有什么，再带着上下文去问，问题才问得到点上。
2. **主动补齐需求**：基于摸底结果，主动推测需求方可能想要的：
   - "项目里已有资产模块，盘点功能是要对现有资产做定期核对吗？"
   - "看到已有导入导出功能，盘点结果需要导出 Excel 吗？"
3. **标注推断**：从背景推测出的需求，标注 `[根据现有功能推断]`，让需求方知道哪些是你的补充。
4. **不卡流程**：信息不全不停止推进。先写能写的，不确定的标 `[待确认]`，集中到阶段 5 再处理。

---

## 待确认项标注法

全流程中遇到不确定的信息，统一用以下标注：

| 标注 | 含义 | 使用时机 |
|------|------|---------|
| `[待确认]` | 需要需求方明确但暂时答不上来 | Gate 3 允许带此标记过门，Gate 5 必须解决 |
| `[根据现有功能推断]` | 基于项目背景推测的需求 | 让需求方知道这是补充的，需确认 |
| `[开放问题]` | 需要进一步讨论或跨部门确认的问题 | 不阻塞文档交付，但必须列在开放问题清单 |
