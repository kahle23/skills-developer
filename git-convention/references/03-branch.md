# 分支操作规范

> 通用安全红线（force/hard/-D 禁止、不自动 push、不提交敏感信息）见 SKILL.md。

## 定位说明

本团队的分支管理以**人工灵活处理**为主。本文档供 AI 理解团队分支约定使用，不要求 AI 自动执行合并/删除等分支操作——涉及分支生命周期时，由用户手动操作或明确指示。

## 分支体系

### 常驻环境分支（长期保留）

| 分支 | 作用 |
|------|------|
| `prod` | 生产环境 |
| `pre` | 预发布环境 |
| `test` | 测试环境 |
| `dev` | 开发环境 |
| `hotfix` | 生产紧急修复 |

这几个分支长期保留，对应特定环境，不删除。

### 临时工作分支（用完即删）

命名规则：

- **版本号命名**：`v1_8_1`、`v1_8_2`，对应某个版本的开发
- **功能描述命名**：`feat_xxxxx`，对应某个功能

临时分支在开发 → 测试 → 发版完成后，会合并到 `prod` 并删除。分支本身不长期保留，开发历史由 commit 记录承载。

> **分支名与 commit 版本号的关系**：两者**不一定相同**。分支名是工作位置标记（如 `v1_8_1`、`feat_xxxxx`），commit 的 version 段是改动所属版本（如 `[fix ][v1.8.1]`）。它们经常巧合一致，但没有强制绑定——例如在 `feat_xxxxx` 分支上提交，commit 仍可能带 `[v1.8.1]`；反过来 `v1_8_1` 分支上的 commit 版本号也可能因切版本而变化。**判定 commit 版本号只看 `git log` 最近提交，不看当前分支名。**

## 合并与清理

- **合并策略**：采用 **rebase**，保持提交历史线性
- **发版后清理**：临时分支合并到 `prod` 后即删除，不堆积旧分支
- **hotfix 同步**：`hotfix` 上的修复合入 `prod` 后，注意同步回 `dev`/`test` 等下游分支，避免修复丢失

> rebase 的具体操作（变基方向、冲突解决）由用户人工判断执行，本规范不提供固定命令模板。

## 分支场景下的注意事项

- ⚠️ rebase 会改写提交历史，对已推送并有人协作的分支 rebase 前务必与团队确认
- ⚠️ 合并到 `prod` 前确认已通过测试
- 分支生命周期操作（合并/删除/切换）默认由用户手动完成；AI 仅在用户明确要求时辅助执行

## Commit Type 与分支的关系

Commit Type（8 种）描述**单次提交做了什么**，分支描述**当前在哪个环境/版本上工作**，两者独立：

- 临时工作分支（如 `v1_8_1`）上可以出现任意 commit type——`[feat]` 为主，穿插 `[fix ]` `[opti]` `[refa]` `[styl]` `[test]` 都正常
- `[doc]` `[misc]` 通常直接在当前工作分支提交，不单独开分支
- 不需要为每种 commit type 对应开分支
