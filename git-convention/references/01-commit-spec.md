# Commit Message 规范

## 格式模板

```
[type][version] 描述信息        ← 有版本（默认，见 COMMIT GATE）
[type] 描述信息                  ← 历史全无版本时的特例
```

- **type 必选**，写法**严格按下方的「Type 类型表」**，表里怎么写就怎么写，不要自己发明规则。
- version 默认带上（判定见下方 Version 节 + SKILL.md COMMIT GATE）。
- ⚠️ "省略 version" 是特例，**仅当最近 5 条 commit 全部无版本**时才用，且需在回复中注明"历史无版本"。

## Type 类型（必选，唯一权威）

> 下表就是 type 的唯一标准写法，**照抄，不要改、不要补、不要删空格**。type 与 version 之间紧邻无空格（即 `][` 中间无空格）。

| 类型 | 含义 | 使用场景 | 判定要点 |
|------|------|----------|---------|
| `[feat]` | 新功能 | 新增业务功能、新接口、新模块 | 有新的对外能力产出 |
| `[fix ]` | 修复 Bug | 修复逻辑错误、异常处理、数据问题 | 修正错误行为，使其符合预期 |
| `[opti]` | 代码优化 | 小幅改进：性能优化、体验改善、局部逻辑调整 | **改动小、不动整体结构**，"打补丁/微调"性质 |
| `[refa]` | 代码重构 | 较大改动：结构调整、模块拆分、职责重新划分 | **改动较大、动整体结构**，但对外行为不变 |
| `[doc]` | 文档 | 文档变更、注释补充 | 只动文档/注释，不碰代码逻辑 |
| `[styl]` | 格式 | 代码格式调整、空格缩进等 | 只动空白/格式，不改逻辑 |
| `[test]` | 测试 | 新增或修改测试用例 | 只动测试代码 |
| `[misc]` | 构建工具 | 构建脚本、依赖更新、配置变更 | 只动构建/依赖/配置，不碰业务代码 |

### `[opti]` 与 `[refa]` 怎么选

两者容易混淆，记住一条：**看改动规模，不看意图好坏**。

| 判据 | `[opti]` 优化 | `[refa]` 重构 |
|------|--------------|--------------|
| 改动规模 | 小（几个文件、几十行内） | 大（跨文件、跨模块） |
| 是否动整体结构 | 否，局部调整 | 是，结构调整 |
| 例子 | 加缓存减少重复查询、提取一个重复字符串为常量 | 拆分大类、调整模块职责、重写整个数据流 |

> 简单记：**小改 = opti，大改 = refa**。拿不准时默认用 `[opti]`。

## Version 版本号（默认带，省略是特例）

> ⚠️ **判定规则见 SKILL.md 的「COMMIT GATE」**。本节为格式细则，判定必须走门禁流程。

**核心原则：版本段默认带上，省略是需要证据的特例。**

- **默认行为**：从最近 commit 历史中取当前版本号。只要最近 5 条 commit 里**任意一条**带版本，本次就**必须带**。
- 分隔符沿用历史：项目首选风格是 `.` 分隔（如 `v1.8.1`）；若最近 commit 用 `_` 分隔（如 `v1_8_1`），则**沿用 `_`**，不混用。
- `v` 一律小写，版本号三位。
- ⚠️ **仅当最近 5 条 commit 全部无版本号**时，才省略 version 段，写 `[type] 描述`。省略时应在回复中注明"历史无版本"，证明你查过。
- ⚠️ **绝不可凭空编造版本号**；也不可"没注意/没跑 log"就默认省略——这是最常见的翻车点。

## 描述信息规范

- **语言**：使用中文
- **长度**：一行不超过 72 个字符
- **内容**：说明"做了什么"，简洁明确
- **禁止**：不要以句号结尾

## 单行优先

- **默认单行**：commit message 尽量只写一行（type + version + 描述），不写多行正文
- **仅在必要时多行**：只有当变更确实重要且一行说不清时（如重大改动、破坏性变更、需要说明背景/影响/迁移方式），才补充多行正文
- **多行格式**：首行为规范标题，空一行后再写正文，正文按条目分行说明

## 示例

有版本（默认，最常见）：

```
[feat][v1.8.1] 资产记录导入：资产类型/子类型匹配数据字典
[fix ][v1.8.1] 修复IT资产二维码生成接口编译错误
[opti][v1.8.0] 优化审批中简述的显示
[refa][v1.8.0] 拆分资产导入服务类，分离解析与持久化逻辑
```

无版本（仅当最近 5 条 commit 全部无版本时才用，需注明）：

```
[doc] 新增资产出入库操作接口说明
```

> 示例中 type 的写法（含空格）严格对照上方「Type 类型表」。
