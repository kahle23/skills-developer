# Git Convention

**本技能性质：Reference 型（规范参考）+ 一个强制门禁。** 大部分内容（type 表、分支命名等）是被动查阅，用的时候翻一下即可；**唯独「COMMIT GATE」是强制门禁**——任何提交都必须先走完，不可凭印象跳过。

团队 Git 操作规范，重点是 Commit Message 格式与提交流程；分支/推送规范作为补充。

## 安全红线

- **禁止自动 push**：commit 后不自动推送，等用户确认
- **禁止暴力操作**：`--force` / `--hard` / `-D` 除非用户明确要求
- **禁止提交敏感信息**：.env、密钥、密码、证书文件不得进入暂存区
- **禁止擅自添加 AI 身份信息**：commit message 严禁附加 AI 生成标记、`Co-authored-by`、`Generated with ...`、署名、emoji 广告等任何 AI/工具身份信息；只写符合规范的变更描述，不改动 author/committer 身份

## 🚧 COMMIT GATE（提交门禁）—— 写 message 前必须通过

这是**强制门禁，不可跳过**。任何 commit message 都必须先走完以下步骤，否则视为违规提交。Reference 型技能不代表可以省略门禁。

### 步骤 1：拉取最近历史

```bash
git log --oneline -10
```

### 步骤 2：显式列出最近提交（在给用户的回复里写出来，不是默记）

把最近 3~5 条 commit 的实际写法**在你的回复文本里列出来**（让用户也能核对你的判定依据），例如：

```
最近提交：
  [opti][v1_8_1] 付款申请单增加模糊查询条件...
  [fix ][v1_8_1] 去掉只有海外才能填写扣款的卡控
  [refa][v1_8_1] Redis ID生成器改为硬编码...
```

### 步骤 3：版本段判定（三选一，必须给结论）

判定**默认值是"带版本"**，省略反而是需要证据的特例：

| 判定 | 条件（严格） | 本次写法 |
|------|-------------|---------|
| ✅ **带版本**（默认） | 最近 5 条 commit 中**任意一条**出现 `[type ][vX_Y_Z]` | 沿用该版本，写 `[type ][v1_8_1]` |
| ✅ **省略版本**（特例） | 最近 5 条 commit **全部**不含 version 段 | 写 `[type ]`，并需在回复中注明"历史无版本" |
| ❌ **禁止凭空编造** | 历史无版本时不得自己补一个版本号 | — |

> **核心规则：只要历史里出现过版本号，就必须带；省略版本才是需要"5 条全部无版本"证据的特例。** 版本的格式细则（分隔符 `.` vs `_`、三位写法等）见 [01-commit-spec.md](references/01-commit-spec.md) Version 节。

### type 写法（严格按 01 Type 表，不发明规则）

- **type 的写法以 `references/01-commit-spec.md` 的「Type 类型（必选）」表为唯一标准**——表里怎么写就怎么写（如 `[feat]`、`[fix ]`、`[doc]`），不要自己补空格或删空格。
- **type 与 version 之间紧邻**，`][` 中间无空格：`[fix ][v1.8.1]` ✓。

完整示例：`[fix ][v1_8_1] 描述`、`[feat][v1_8_1] 描述`（type 部分照 01 表，版本部分紧邻接上）。

### 反面案例（真实翻车记录）

> 2026-07-23 会话：AI 没跑 `git log` 就提交 `[fix ] 付款发票校验方法补充事务控制`，漏掉版本段。实际本仓库历史**普遍带版本**（`[opti][v1_8_1]`），用户当场指出。后 amend 补成 `[fix ][v1_8_1] 付款发票校验方法补充事务控制`。
> 教训：**不跑 log = 必然翻车**。门禁步骤 1 不可省略。

### 正面示范

✅ 跑了 log → 看到 `[opti][v1_8_1]`、`[fix ][v1_8_1]` → 判定"带版本 v1_8_1" → 写 `[fix ][v1_8_1] 修复资产导入空指针问题`
✅ 跑了 log → 5 条全是 `[doc] xxx`、`[misc] xxx` 无版本 → 判定"历史无版本" → 写 `[doc] 更新资产模块接口说明`（并在回复里注明）
❌ 没跑 log 就提交（必翻车）
❌ 跑了 log 但跳过判定，直接写 `[fix ] 修复xxx`（漏版本）

### ⚡ commit 前一秒自查清单

打 commit 命令前，逐条默念；**任一条不过 = 暂停，回去补**：

- [ ] 跑过 `git log --oneline -10` 并在回复里列出了最近提交？
- [ ] 版本段判定：历史带版本 → 本次**带**（已填入具体版本，如 `v1_8_1`）；历史全无版本 → 省略并已注明？
- [ ] type 写法严格对照 01「Type 类型表」（不自己补/删空格），`][` 之间紧邻无空格？
- [ ] message 里**没有** AI 身份信息（无 `Co-authored-by`、无署名、无 emoji 广告）？
- [ ] 只 `git add` 了本次相关文件，没用 `git add -A` / `.`？
- [ ] commit 前看过 diff，确认无敏感信息（密钥/密码/.env）？
- [ ] commit 后**不自动 push**（等用户确认）？

## 团队分支约定（与通用 Git Flow 的关键差异）

本团队的分支管理有几条**与通用 Git Flow 不同**的做法，AI 必须了解，完整规范见 [03-branch.md](references/03-branch.md)：

1. **分支按"环境/版本"命名，不用 `feature/` `fix/` 前缀**——常驻环境分支（prod/pre/test/dev/hotfix）+ 临时分支（`v1_8_1`、`feat_xxxxx`）
2. **合并用 rebase，不用 merge**——保持提交历史线性
3. **临时分支用完即删**——发版合并到 `prod` 后删除，开发历史由 commit message 承载
4. **分支操作以人工灵活处理为主**——默认由用户手动操作，AI 仅在明确要求时辅助

> Commit Type（描述单次提交做了什么）与分支（描述当前在哪个环境/版本工作）是独立的两个维度，不绑定——详见 [03-branch.md](references/03-branch.md)「Commit Type 与分支的关系」。
