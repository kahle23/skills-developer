# Commit Message 规范

## 格式模板

```
[type][version] 描述信息
```

- type 为必选，version 和描述信息按规则填写
- **无版本号时**：省略 version 段，格式变为 `[type] 描述信息`

## Type 类型（必选）

| 类型 | 含义 | 使用场景 |
|------|------|----------|
| `[feat]` | 新功能 | 新增业务功能、新接口、新模块 |
| `[fix ]` | 修复 Bug | 修复逻辑错误、异常处理、数据问题 |
| `[opti]` | 优化改进 | 性能优化、代码重构、体验改善 |
| `[refa]` | 重构 | 不改变功能的代码结构调整 |
| `[doc]` | 文档 | 文档变更、注释补充 |
| `[styl]` | 格式 | 代码格式调整、空格缩进等 |
| `[test]` | 测试 | 新增或修改测试用例 |
| `[misc]` | 构建工具 | 构建脚本、依赖更新、配置变更 |

> type 标记统一占 4 字符位以保持对齐。`[fix ]` 中的尾部空格是规范要求，实际 commit message 中也写 `[fix ]`，不是笔误。

## Version 版本号

- 从最近的 commit 历史中获取当前版本号，保持一致
- `v` 一律小写，版本号三位，分隔符用 `.` 或 `_`
- 格式示例：`[v1.8.1]`、`[v1_8_0]`
- ⚠️ **如果最近 commit 历史中没有版本号**，直接省略 version 段，不要自行编造版本号

## 描述信息规范

- **语言**：使用中文
- **长度**：一行不超过 72 个字符
- **内容**：说明"做了什么"，简洁明确
- **禁止**：不要以句号结尾

## 单行优先

- **默认单行**：commit message 尽量只写一行（type + version + 描述），不写多行正文
- **仅在必要时多行**：只有当变更确实重要且一行说不清时（如重大改动、破坏性变更、需要说明背景/影响/迁移方式），才补充多行正文
- **多行格式**：首行为规范标题，空一行后再写正文，正文按条目分行说明

## 示例

```
[feat][v1.8.1] 资产记录导入：资产类型/子类型匹配数据字典
[fix ][v1.8.1] 修复IT资产二维码生成接口编译错误
[opti][v1.8.0] 优化审批中简述的显示
[feat] 新增资产出入库操作的接口
```

最后一条示例为无版本号的写法。
