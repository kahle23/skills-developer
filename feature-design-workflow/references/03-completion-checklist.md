# 03 — 完成度清单（Definition of Done）

设计文档定稿前，逐项过这份清单。分为四组，每组全部打勾才算该组通过。

---

## 一、格式规范

- [ ] 文档保存到 `docs/` 目录，文件名建议 `编号-模块名.md`（无编号体系时省略编号）
- [ ] 文档使用中文编写
- [ ] SQL 代码块完整可执行（不是伪代码、不是省略号）
- [ ] 接口地址与表名前缀保持一致
- [ ] 如存在历史设计文档，格式和风格保持一致

## 二、完整性

- [ ] 五个章节全部输出（需求概述 / 功能模块梳理 / 表结构设计 / 接口梳理 / 关键业务逻辑）
- [ ] 每个子模块都标注了"核心/扩展"
- [ ] 每个功能点都有对应接口（或标注"前端控制"）
- [ ] 新建表和复用表都列出了
- [ ] 有枚举/状态字段的，取值含义已标注

## 三、质量

- [ ] 需求背景说清了"为什么做"，不是复述功能名
- [ ] 功能点描述具体，不是"管理 XX"这种泛词
- [ ] 接口核心入参列全，不是"等字段"
- [ ] 关键逻辑（编码规则/状态流转/权限控制）有明确规则，不是"按业务需要处理"
- [ ] 文档中没有遗留的 TODO 或空表格

## 四、交叉验证

- [ ] 接口列表的功能覆盖了模块梳理中的功能点（无遗漏）
- [ ] 表结构字段支持了接口的入参需求（无字段缺失）
- [ ] 状态流转规则与字段取值定义一致（如字段定义 3 个状态，流转图不能出现第 4 个）
- [ ] 数据权限控制覆盖了主要角色
- [ ] 所有 `[待确认]` 项已解决，或已汇总为待跟进清单

---

## 使用方式

1. 文档初稿完成后，逐组过清单
2. 未通过项标注原因（缺信息 / 需确认 / 需补充）
3. 全部通过 → 可以进入 Gate 4 交付确认
4. 有未通过项 → 回阶段 3 补齐，或标注为已知缺口后与开发者确认是否可接受
