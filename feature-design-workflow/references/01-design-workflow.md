# 01 — 设计工作流

本工作流分 4 个阶段、3 道门禁（Gate）。**每个阶段的 Gate 检查通过后才能进入下一阶段。**

如果 Gate 没过，就停在这一步补齐，不要往后跑。

---

## 阶段 1：需求理解

**目标**：搞清楚要做什么、不做什么、边界在哪。

**动作**：

与开发者沟通，确认以下信息：

- 功能背景与目标（为什么要做这个功能）
- 功能范围（包含哪些子模块、不包含什么）
- 核心业务流程与边界条件
- 特殊场景（异常流程、权限边界）

> 如果用户提供了原型图/截图，先分析图片内容再提问。

### Gate 1：需求边界确认

**过门检查清单**（逐项确认，全部打勾才放行）：

- [ ] 能用一句话说清这个功能做什么
- [ ] 能列出"不做什么"（至少 2 项排除项）
- [ ] 核心业务流程的起点和终点明确
- [ ] 至少识别出 1 个边界条件或异常场景
- [ ] 有业务专属术语的，已定义清楚

**没过 Gate 1 的典型情况**：

- 开发者说"做个 XX 管理"但说不清范围 → 追问"除了 XX 还管什么？不管什么？"
- 没有边界条件 → 主动问"如果 XX 为空怎么办？""XX 不存在时怎么处理？"
- 开发者也不确定 → 先记录为 `[待确认]`，但必须标注清楚，Gate 1 允许带 `[待确认]` 过门，Gate 3 不允许

---

## 阶段 2：现状摸底

**目标**：搞清楚项目里已有什么，避免重复造轮子和风格跑偏。

**动作**：

**2a. 查看历史文档风格**

如果项目存在历史设计文档（如 `docs/` 目录下的 `.md` 文件），优先读取并分析：

- 文档章节结构和排版风格
- 表格格式和字段描述方式
- 接口地址命名模式
- SQL 格式化风格

> 新文档的风格必须与历史文档保持一致。

**2b. 查看现有表结构**

查找项目的 SQL/表结构文件（通常在 `docs/`、`sql/`、`资源目录` 下，或数据库脚本中）：

- 了解模块前缀惯例（如 `oa_`、`crm_`）
- 确认已有的标准字段风格
- 识别可复用的现有表

### Gate 2：现状可设计确认

**过门检查清单**：

- [ ] 已查看（或确认没有）历史设计文档，知道目标风格
- [ ] 已了解表命名前缀惯例
- [ ] 已识别可复用的现有表（或确认无）
- [ ] 如果项目有历史文档但没找到 → 已向开发者确认"是否需要参考某份文档"

**没过 Gate 2 的典型情况**：

- 项目有历史文档但没找到 → 问开发者"之前的设计文档放在哪了？"
- 找到多个风格不同的历史文档 → 问"以哪份为准？"

---

## 阶段 3：文档编写

**目标**：按模板输出完整设计文档。

**动作**：

按 `02-doc-template.md` 中的模板输出完整文档，包含全部五个章节。

模板里每章末尾都有 **mini-gate**（章节级检查），写完一章就过一遍。

### Gate 3：文档完整性检查

**过门检查清单**：

- [ ] 五个章节全部输出（需求概述 / 功能模块梳理 / 表结构设计 / 接口梳理 / 关键业务逻辑）
- [ ] 每章的 mini-gate 检查全部通过
- [ ] 所有 Gate 1 中标记的 `[待确认]` 项已解决或已标注为需跟进
- [ ] 表结构 SQL 完整可执行
- [ ] 接口分组清晰，核心入参列全

**没过 Gate 3 的典型情况**：

- `[待确认]` 项还没解决 → 列出来让开发者逐个确认，或明确标注为文档缺口
- 某章节 mini-gate 没过 → 回去补那一章

---

## 阶段 4：交付确认

**目标**：文档定稿，确认可以进入开发。

**动作**：

1. 按 `03-completion-checklist.md` 逐项过完成度清单
2. 向开发者展示检查结果，标注已通过项和待处理项
3. 根据反馈迭代调整

### Gate 4：交付确认

**过门检查清单**：

- [ ] 完成度清单四组检查全部通过（格式规范 / 完整性 / 质量 / 交叉验证）
- [ ] 开发者确认文档可以指导开发
- [ ] 文档已保存到正确位置（`docs/` 目录）

---

## 草台班子速查：需求只有截图时怎么办

团队实际场景：需求来源是一张原型截图 + 几句口头沟通，没有正式的需求文档。

**应对策略**：

1. **先分析截图**：识别页面元素（表格/表单/按钮/状态标签），反推功能点
2. **主动补齐需求**：截图能看出来但不明确的地方，主动问而不是等用户说
   - "截图里设备状态有 4 种，是否有其他状态？"
   - "这个列表是否需要批量操作？"
   - "详情页截图没给，是弹窗还是新页面？"
3. **标注重建需求**：从截图推断出的需求，在文档中标注 `[根据原型推断]`，让开发者知道哪些是明确需求、哪些是你的补充
4. **不卡流程**：不要因为信息不全就停下来反复追问。先把能写的写了，不确定的地方标 `[待确认]`，继续推进——等 Gate 3 再集中处理
