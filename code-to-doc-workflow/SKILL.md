# 代码转文档工作流

从**已有代码 + 截图 + 口头描述**逆向生成文档。专为"需求靠截图和口头沟通、文档严重滞后"的团队设计。

## 本技能性质

这是 **Workflow 型技能**（流程引导）。核心动作是**端到端跟做**——引导你走完从读代码到输出文档的完整流程，每个阶段有过门检查（Gate），过门才进下一步。与 Reference 型技能（查阅规范）不同，本技能不需要从头读，跟着流程走就行。

一份代码，同时产出两类文档：

- **技术文档**（给开发看）：技术设计、业务流程、接口、数据存储、关键逻辑
- **用户使用文档**（给最终业务用户看）：傻瓜式操作步骤、前置条件、常见问题

## 适用场景

> 团队是"草台班子"模式：需求 = 截图 + 口头沟通 + 简单文档 → 直接开发 → 文档缺失。
> 本技能解决"代码已经写完了，但文档从零开始"的痛点。

核心思路：**代码即真相**。当代码与口头描述冲突时，以代码实际行为为准。

## 流程总览

```
阶段1 收集输入 → Gate1 → 阶段2 读代码 → Gate2 → 阶段3 输出文档 → Gate3
```

每个 Gate 都有过门检查清单，详见 `01-workflow.md`。

## 加载规则

| 用户意图 / 当前状态 | 读取 |
|---------|------|
| 开始梳理/生成文档 | 先读 01，进入阶段 1 |
| 在读代码阶段卡住 / 代码量大不知从哪入手 | 04 |
| 要输出文档了，不确定章节怎么写 | 02（技术文档）或 03（用户文档），看 99 对照样例 |
| 觉得文档差不多了，想检查 | 01 的 Gate3 检查清单 |
| 看完整范例对照写法 | 99 |

## 关联技能

- `feature-design-workflow` — 功能**设计**文档（开发前写）。本技能是开发**后**从代码逆向，方向相反，互补使用。
- `java-coding-standard` — Java 编码规范，阅读后端 Java 代码时辅助理解框架约定（如 Baibao/Kunlun 的 Service 分层）。
- `db-design-standard` — 表结构设计规范，技术文档中"数据存储"章节涉及表结构时参考其命名/字段规范。
- `code-to-test-workflow` — 同属"代码逆向"系列（文档 vs 测试），互补使用。
