# 03 - 用户使用文档模板

> 给**最终业务用户**看的操作指南。用户照着这份文档，不看代码就能用起来。

## 写作原则

1. **说人话**：不出现"Controller""Service""接口""数据库"等技术词
2. **步骤化**：操作用编号步骤，一步只做一件事
3. **截图为王**：每个关键步骤配截图，文字是辅助
4. **预设用户是新手**：默认用户没用过类似系统，必要的概念要解释
5. **前置条件必写**：用户开始操作前需要准备什么、权限是什么

---

## 模板正文

### {功能名称} - 使用说明

#### 这个功能是干什么的？

> 用一两句话说清功能价值，站在用户角度。

{例如：资产领用功能让您在线申请领用公司资产（如电脑、办公设备），提交后由管理员审批，通过后即可领取。}

#### 开始之前

> ⚠️ 前置条件必须写清楚，否则用户走到一半才发现没权限/没准备。

使用本功能前，请确认：

- [ ] 您有系统登录账号，且有 {功能} 的使用权限
- [ ] {其他前置条件，如：您的部门已在系统中配置}

> 如果不确定自己有没有权限，请联系系统管理员 {联系方式占位}。

## 截图标注规范

截图不是简单贴图，要**引导用户视线**：

1. **圈出关键区域**：用红色方框圈出按钮、菜单、输入框
2. **箭头指引**：操作顺序用数字标记（①②③）
3. **占位格式**：统一用 `> 📷 *[截图：描述]*`，方便后续统一替换
4. **重点加粗**：截图描述中，按钮名、菜单名加粗

示例：
```markdown
> 📷 *[截图：在系统主界面，用红框标注左侧菜单的「资产管理」]*
```

## 把技术逻辑翻译成用户语言

代码里的概念，用户文档里要转换：

| 代码/技术概念 | 用户文档表述 |
|--------------|-------------|
| status=0 | "待审批"（不提状态码） |
| 接口返回错误码 40001 | "提示：资产已被领用" |
| 数据库唯一约束冲突 | "该记录已存在，请勿重复提交" |
| Controller → Service | （完全不提，只说操作结果） |
| 并发锁 | （不提，用户只关心"能不能操作"） |

## 待补充清单

> 用户文档中需要人工补充的，主要是截图和业务确认。

| 序号 | 内容 | 说明 |
|------|------|------|
| 1 | 各步骤截图 | 代码无法生成截图，需要人工截取 |
| 2 | 菜单/按钮实际名称 | 代码中的命名和界面上显示的可能不同 |
| 3 | 联系方式 | 管理员/支持的联系方式 |
| 4 | 权限规则 | 哪些角色能用、哪些不能，需业务方确认 |
