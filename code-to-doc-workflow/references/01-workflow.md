# 01 — 工作流：从代码到文档

本工作流分 3 个阶段、3 道门禁（Gate）。**每个阶段的 Gate 检查通过后才能进入下一阶段。**

核心原则：**代码即真相**。代码与口头描述冲突时，以代码实际行为为准，但注明差异。

---

## 阶段 1：收集输入

**目标**：拿到足够的材料开始工作。

**动作**：

| 输入类型 | 说明 | 必需性 |
|---------|------|--------|
| 代码路径（前端/后端） | 哪个模块、哪些目录 | ⚠️必需，至少有其一 |
| 截图 | 需求原型图、UI 截图、系统界面截图 | 推荐，帮助理解业务意图 |
| 口头描述 | 功能是干什么的、谁用、什么场景 | 推荐，加速理解 |
| 数据库访问 | 能看表结构最好 | 可选，有则更准 |

### Gate 1：输入充足确认

**过门检查清单**：

- [ ] 至少拿到了前端或后端代码路径
- [ ] 知道这个功能大概干什么（用户说了，或能从代码推断）

**没过 Gate 1 的典型情况**：

- 用户只说了"帮我梳理文档"但没给代码路径 → 问"哪个模块？代码在哪？"
- 代码路径拿到了但完全不知道功能是啥 → 不追问，直接读代码反推（草台班子模式，Gate 1 放行）

> **草台班子场景**：用户可能只甩来一句"帮我梳理 XX 功能的文档"加代码路径，没有截图也没有描述。这时**不要追问**，直接进入阶段 2，缺口等 Gate 3 集中处理。

---

## 阶段 2：读代码

**目标**：从代码中还原出业务全貌——这个功能解决什么问题、数据怎么流动、有哪些业务规则。

**动作**：

按 `04-code-reading-strategy.md` 进行。核心原则：

- **先找入口**（Controller / 路由 / 页面组件），不要从底层工具类开始读
- **顺调用链**：入口 → 业务逻辑 → 数据操作
- **看命名反推业务**：变量名、方法名、表名往往暴露业务语义

读完后，在脑中建立三件事：

1. **这个功能解决什么问题**（从命名、注释、业务逻辑推断）
2. **数据怎么流动**（前端表单 → 接口 → Service → 数据库）
3. **有哪些业务规则**（校验、状态流转、权限控制）

### Gate 2：代码已读懂确认

**过门检查清单**：

- [ ] 能说清这个功能的核心流程（数据从哪来、到哪去）
- [ ] 找到了所有入口（Controller / 页面组件），没有遗漏
- [ ] 核心业务规则已提取（校验规则、状态流转、权限控制）
- [ ] 前后端接口对应关系已梳理（哪个前端操作调哪个后端接口）

**没过 Gate 2 的典型情况**：

- 只读了前端或后端，另一端代码找不到 → 标注"需要 XX 端代码补充"，但已读部分可以继续
- 代码量太大读不完 → 先列文档大纲，确认核心模块后再逐章填充
- 某段代码完全看不懂 → 标注 `[待确认：XX 逻辑]`，不卡流程，继续往下走

---

## 阶段 3：输出文档

**目标**：按模板输出完整文档，缺口全部标注。

**动作**：

**3a. 选模板**：

| 要什么 | 用哪个模板 |
|--------|-----------|
| 技术文档（给开发看） | 02-技术文档模板 |
| 用户使用文档（给最终用户看） | 03-用户使用文档模板 |
| 两个都要 | 两个都输出，用户文档基于技术文档的内容转化 |

**3b. 识别并标注缺口**：

代码读不出来的东西，必须识别出来：

| 缺口类型 | 例子 | 处理方式 |
|---------|------|---------|
| 业务背景 | 为什么做这个功能 | 文档中标注 `[待补充：业务背景]` |
| 特殊场景 | 某些边界条件的设计意图 | 标注 `[待确认：XX 场景的处理逻辑]` |
| 非功能需求 | 性能要求、并发量 | 标注 `[待补充：性能要求]` |
| 历史原因 | 为什么这么实现 | 不写，或标注 `[实现原因待确认]` |

**3c. 标注格式**：

```markdown
<!-- 待补充：这里需要业务方确认审批流程的具体规则 -->
[待补充：审批节点和审批人规则]

<!-- 待确认：代码中这里是写死的，不确定是否有动态配置 -->
[待确认：超时时间是否需要可配置]
```

### Gate 3：文档完整性检查

**过门检查清单**：

- [ ] 文档类型选对了（技术文档 vs 用户文档）
- [ ] 业务流程能自洽（数据从哪来、到哪去、怎么流转）
- [ ] 核心接口都在（技术文档）/ 核心操作步骤都在（用户文档）
- [ ] 所有不确定的地方都标注了 `[待补充]` 或 `[待确认]`
- [ ] 用户文档的操作步骤，用户照着做能走通
- [ ] 文档末尾有待补充清单

---

## 草台班子速查

| 情况 | 处理 |
|------|------|
| 只有代码路径，没截图没描述 | 直接读代码反推，不追问，输出后标注缺口 |
| 口头描述和代码冲突 | ⚠️**以代码为准**，但注明"口头描述为 XX，代码实际为 YY，请确认" |
| 代码不完整（只有前端或只有后端） | 读到什么写什么，缺口标注"需要 XX 端代码补充" |
| 功能很大，代码很多 | 先列文档大纲，确认后再逐章填充 |
| 截图和代码对不上 | 以代码为准，注明差异 |
