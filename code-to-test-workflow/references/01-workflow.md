# 01 — 工作流：从代码到测试

本工作流分 3 个阶段、3 道门禁（Gate）。**每个阶段的 Gate 检查通过后才能进入下一阶段。**

核心原则：**代码即真相**。从代码中提取表单字段、校验规则、业务流程、状态流转，自动生成覆盖正常/异常/边界的测试用例和对应脚本。

---

## 阶段 1：收集输入

**目标**：拿到足够的材料开始工作。

**动作**：

| 输入类型 | 说明 | 必需性 |
|---------|------|--------|
| 代码路径（前端/后端） | 哪个模块、哪些目录 | ⚠️必需，至少有其一 |
| 截图 | 页面 UI 截图，帮助理解表单结构和交互 | 推荐，加速用例设计 |
| 口头描述 | 功能是干什么的、谁用、什么场景 | 推荐，加速理解 |
| 系统访问地址 | 测试环境的 URL、登录账号 | ⚠️生成脚本时必需 |
| 数据库访问 | 能看表结构和数据，用于断言验证 | 可选，有则断言更准 |

### Gate 1：输入充足确认

**过门检查清单**：

- [ ] 至少拿到了前端或后端代码路径
- [ ] 知道这个功能大概干什么（用户说了，或能从代码推断）
- [ ] 如果要生成脚本 → 有测试环境地址（没有的话标注 `[待补充]` 后可放行）

**没过 Gate 1 的典型情况**：

- 用户只说了"帮我写测试"但没给代码路径 → 问"哪个模块？代码在哪？"
- 要生成脚本但没有测试环境地址 → 不追问，进入阶段 2，地址标 `[待补充：测试环境 URL]`

> **草台班子场景**：用户可能只甩来一句"帮我给 XX 功能写测试"加代码路径。这时**不要追问一堆**，先读代码生成用例和脚本框架，环境信息标注为待补充。

---

## 阶段 2：读代码 + 提取测试点

**目标**：从代码中提取所有可测试的点。

**动作**：

**2a. 读代码（测试视角）**

按 `04-code-reading-strategy.md` 进行。核心原则：

- **前端找测试点**：表单字段、校验规则（required/pattern/validator）、按钮状态（disabled 条件）、交互行为
- **后端找校验逻辑**：Controller 入参校验、Service 业务校验、状态流转条件
- **前后端对应**：前端表单字段 → 后端接口参数 → 数据库字段，三者对应起来就是断言依据

**2b. 提取测试点**

读代码后，列出所有可测试的点：

| 测试点类型 | 怎么提取 | 例子 |
|-----------|---------|------|
| 表单必填校验 | 前端 `rules` 的 `required: true` | "用户名"为空时提交应报错 |
| 格式校验 | 前端 `pattern` 或自定义 `validator` | 手机号格式不对应报错 |
| 业务校验 | 后端 Service 的 if/throw | 重复编码应提示"编码已存在" |
| 正常流程 | 主流程调用链 | 填完所有字段提交应成功 |
| 状态流转 | 后端 status 字段的变更 | 提交后状态变为"待审核" |
| 权限控制 | `@PreAuthorize` / 按钮 disabled | 无权限应看不到按钮 |
| 边界值 | 字段长度限制、数值范围 | 名称超过 50 字应报错 |
| 交互联动 | 级联选择、联动表单 | 选了部门后岗位列表应刷新 |

### Gate 2：测试点提取完整确认

**过门检查清单**：

- [ ] 前端表单的所有字段都列出了（包括类型、校验规则）
- [ ] 后端核心校验逻辑都找出了（Controller + Service 层）
- [ ] 前后端接口对应关系已梳理（哪个前端操作调哪个接口）
- [ ] 状态流转规则已提取（如有 status 字段）
- [ ] 测试点列表中，每个测试点都能说清"测什么、怎么测"

**没过 Gate 2 的典型情况**：

- 只读了前端或后端 → 已读部分继续，另一端标注"需要 XX 端代码补充"
- 前端框架不确定（Vue? React? jQuery?）→ 按 04 的「前端框架自动检测」流程判断
- 校验规则太多列不完 → 先覆盖核心字段和所有校验规则，次要字段用参数化批量测
- 代码完全看不懂某段 → 标注 `[待确认]`，不卡流程

---

## 阶段 3：输出用例 + 脚本

**目标**：按模板输出完整的测试用例文档 + Playwright 脚本。

**动作**：

**3a. 设计测试用例**

按 `02-test-case-template.md` 输出。关键原则：

- **每个测试点至少一个用例**
- **正常路径 + 异常路径 + 边界值都要覆盖**
- **预期结果必须可验证**（能看到的东西：提示消息、页面跳转、数据变化、按钮状态）

**3b. 生成 Playwright 脚本**

按 `03-script-template.md` 和 `05-interactions-cookbook.md` 生成。关键原则：

- **脚本和用例一一对应**：用例编号写在测试函数的 docstring 里
- **用 POM（页对象模式）**：页面元素定位和操作封装成页类，不散落在测试里
- **操作细节查 05**：下拉怎么选、日期怎么填、文件怎么传，全在交互手册里

**3c. 标注待补充项**

```markdown
<!-- 待补充：测试环境地址 -->
[待补充：测试环境 URL]

<!-- 待确认：不确定这个字段是否有唯一性校验 -->
[待确认：编码字段是否做了后端唯一性校验]
```

脚本中用 `# TODO` 标注：

```python
# TODO: 替换为实际测试环境地址
BASE_URL = "http://localhost:8080"
```

### Gate 3：用例+脚本完整性检查

**过门检查清单**：

- [ ] 测试用例覆盖了正常路径（至少 1 个主流程用例）
- [ ] 测试用例覆盖了异常路径（校验失败、权限不足、重复提交）
- [ ] 测试用例覆盖了边界值（空值、最大长度、临界数值）
- [ ] 每个用例的预期结果都可验证（不是"应该成功"，而是"提示'新增成功'"）
- [ ] 脚本的测试函数和用例编号一一对应
- [ ] 元素定位用的是推荐方式（get_by_label / get_by_role 优先）
- [ ] 交互操作正确（原生 select 用 select_option，UI 组件库下拉用 click + click）
- [ ] 不确定的地方都标注了 `[待补充]` / `[待确认]` / `# TODO`
- [ ] 末尾有待补充清单

## 草台班子速查

| 情况 | 处理 |
|------|------|
| 只有代码路径，没有截图和环境 | 先读代码生成用例和脚本框架，环境信息和截图标注为待补充 |
| 口头描述和代码冲突 | ⚠️**以代码为准**，但注明"口头描述为 XX，代码实际为 YY，请确认" |
| 只有前端代码 | 从前端表单和 API 调用生成用例，后端校验逻辑标注"需要后端代码补充" |
| 只有后端代码 | 从 Controller 和 Service 生成接口测试用例，UI 操作步骤标注"需要前端代码或截图补充" |
| 功能很大，代码很多 | 先列测试用例大纲（测试点清单），确认后再逐个细化 |
| 表单字段太多 | 先覆盖核心字段和所有校验规则，次要字段用参数化批量测 |
