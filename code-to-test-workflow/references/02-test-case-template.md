# 02 - 测试用例模板

> 给**测试人员/开发**看的结构化测试用例文档。基于代码逆向生成，确保功能的核心路径都被覆盖。

## 用例编号规则

```
TC-{模块缩写}-{三位序号}

示例：
TC-USER-001    用户模块第 1 个用例
TC-USER-002    用户模块第 2 个用例
TC-ASSET-001   资产模块第 1 个用例
```

模块缩写从代码目录名或功能名提取，全大写。

---

## 覆盖维度

每个功能至少覆盖以下维度：

| 维度 | 占比建议 | 说明 |
|------|---------|------|
| 正常路径（Happy Path） | ~30% | 按设计流程走通，验证功能可用 |
| 异常路径 | ~40% | 校验失败、权限不足、重复操作、网络错误 |
| 边界值 | ~20% | 空值、最大长度、临界数值、特殊字符 |
| 交互联动 | ~10% | 级联选择、联动刷新、状态切换 |

## 优先级判定

| 优先级 | 标准 | 示例 |
|--------|------|------|
| **P0** | 核心功能、主流程、阻塞级 bug | 正常新增、必填校验、核心业务校验 |
| **P1** | 重要功能、常见异常路径 | 唯一性校验、格式校验、权限控制 |
| **P2** | 边界值、不常见场景 | 超长字符串、特殊字符、并发提交 |
| **P3** | UI 细节、优化体验 | 按钮 hover 效果、输入框 placeholder |

---

## ⚠️ 预期结果写作要点

> 预期结果必须**可验证**——能在界面上看到、能在网络请求中看到、能在数据库中看到。

| ❌ 不可验证 | ✅ 可验证 |
|------------|----------|
| 应该成功 | 页面提示 `新增成功`（可见的 toast/message） |
| 报错了 | "手机号"输入框下方显示 `手机号格式不正确` |
| 不能提交 | 点击保存后，Network 中无 POST 请求 |
| 数据正确 | 列表中新增一行，用户名为 `testuser01` |

验证来源优先级：

1. **页面可见元素**（提示消息、列表数据、按钮状态）— 最推荐
2. **网络请求**（响应状态码、响应体内容）— 接口测试时用
3. **数据库**（表记录变化）— 需要数据库访问权限，断言最强

## 异常用例设计清单

从代码中提取以下信号，生成对应的异常用例：

| 代码中的信号 | 对应的异常用例 |
|-------------|--------------|
| 前端 `required: true` | 该字段为空时提交 → 应报错 |
| 前端 `pattern: /regex/` | 输入不匹配格式 → 应报错 |
| 前端 `max: 50` | 输入超长字符串 → 应报错 |
| 后端 `if (exists) throw` | 输入已存在的唯一值 → 应报"已存在" |
| 后端 `@PreAuthorize` | 无权限角色操作 → 应被拒绝 |
| 按钮 `:disabled="!form.valid"` | 表单不满足条件时按钮应不可点 |
| 状态字段 `if status != XX` | 非预期状态下操作 → 应报状态错误 |
