# 01 — 设计工作流

本文档覆盖两类场景：**设计新表**（第 1-5 步）和 **修改已有表结构**（见末尾）。

## 一、设计新表

### 第 1 步：理解需求

明确以下信息：
- 要存储什么数据（实体是什么）
- 数据之间的关系（一对一 / 一对多 / 多对多）
- 核心业务字段有哪些
- 查询场景（高频查询字段决定索引设计）
- 是否多租户场景（决定 `tenant_id` 是否纳入组合索引）

### 第 2 步：查看历史风格（有兜底）

如果项目存在 `docs/table.sql`、`sql/` 目录或类似的历史 DDL 文件，**优先读取并分析**：
- 模块前缀惯例（如 `oa_`、`crm_`）
- 已有的标准字段风格
- 索引命名模式
- SQL 格式化风格

> 新表的风格必须与项目已有表保持一致。

⚠️ **兜底策略**：如果项目中没有任何历史 DDL 文件（新项目 / 空仓库），则直接按 `03-table-standards.md` 的默认规范设计，并在输出时说明"未发现历史 DDL，采用默认风格，请确认是否与团队约定一致"。

### 第 3 步：输出设计方案概要（确认闭环）

在写完整 DDL 之前，先输出**简明设计概要**供用户确认，避免返工：

```
设计概要：
- 表名：oa_xxx_xxx
- 核心字段：（列出关键业务字段及类型）
- 索引：（列出计划的索引及理由）
- 关联表：（列出逻辑外键指向哪张表）
```

用户确认后再进入第 4 步。如果用户明确说"直接出 SQL"或场景非常简单，可跳过此步。

### 第 4 步：生成 DDL

按 `03-table-standards.md` 第 5 节的 SQL 输出格式，生成完整的 CREATE TABLE 语句。输出前对照 `02-examples.md` 确认格式无误。

### 第 5 步：补充说明

输出表结构后，同时提供：
1. **索引设计理由** — 为什么建这些索引（支撑哪些查询场景）
2. **字段填写说明** — 哪些必填、哪些有默认值、特殊字段的业务含义
3. **与现有表的关联关系** — 外键逻辑关联（不强制物理外键）

---

## 二、修改已有表结构

修改场景（新增字段 / 删除字段 / 修改字段类型 / 加索引 / 改索引）的流程：

1. **先读取目标表的现有 DDL**：通过 `SHOW CREATE TABLE` 或项目的 `*.sql` 文件了解当前结构
2. **对照 03 规范**：新增的字段命名、类型选择、索引命名必须符合规范
3. **评估影响**：新增字段是否影响已有数据？是否需要回填默认值？加索引是否锁表（大表注意）？
4. **输出 ALTER 语句**：
   ```sql
   ALTER TABLE `表名`
     ADD COLUMN `新字段` varchar(100) NOT NULL DEFAULT '' COMMENT '说明' AFTER `某字段`;
   ```
5. **补充影响说明**：回填策略、索引构建耗时预估、兼容性提示

> 修改表结构同样遵守 03 的命名规范和禁止项，不允许因为"随手改"而破坏一致性。
